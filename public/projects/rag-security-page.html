Defense Architecture
5 layers. Every query runs through all of them.
No query skips a layer. Attacks blocked early save compute. Safe queries flow through to a grounded, access-controlled response.
L1
Input Validation
~1ms
Regex pattern matching blocks known injection phrases before the LLM ever sees the query. Zero compute cost, zero latency.
Patterns: "ignore instructions" · "system prompt" · "jailbreak" · "bypass" · base64 obfuscation · role-injection strings
✗ BLOCKif regex match → immediate rejection, logged to audit
✓ PASS→ Layer 2
L2
LLM Classifier
1–2s
llama-3.3-70b judges jailbreak intent with a structured JSON response. Catches semantic attacks that regex misses — rewording, euphemisms, indirect escalation.
Returns: {"is_malicious": bool, "confidence": 0–1, "reason": "..."}
Blocks if confidence > 0.70
✗ BLOCKif confidence > 70% → blocked with reason logged
✓ PASS→ Layer 3
L3
Hybrid Retrieval + RBAC
100–500ms
Dense (ChromaDB cosine) + Sparse (BM25) retrieval, merged via Reciprocal Rank Fusion, then cross-encoder reranked. RBAC filter strips any document above the user's tier before the LLM sees it.
ChromaDB cosine · BM25Okapi · RRF (k=60) · ms-marco cross-encoder · role mask: public / employee / admin · sensitive values redacted before prompt assembly
∅ no_docsif role lacks access → "no relevant documents" (not a refusal)
✓ PASStop-5 chunks → LLM generates answer → Layer 4
L4
Response Filter
1–2s
LLM-as-judge scans the generated response for leaked secrets, PII, or out-of-scope data before it leaves the system. Last line of defense against context-leakage attacks.
Checks for: credential patterns · PII (names, emails, SSNs) · data above user tier · prompt echo / system instruction leakage
✗ REDACTif unsafe content detected → response redacted, incident logged
✓ PASS→ Layer 5 (audit) → response returned
L5
Audit Logger
<1ms
Every query — whether blocked, denied, or answered — is logged with full context. Complete incident trail for forensics and compliance.
Logs: timestamp · query hash · user role · layer reached · block reason · response fingerprint · latency per layer
✓ ALWAYSruns unconditionally — attacks and safe queries both logged

Key Findings
What the evaluation reveals
🛡️
100% block rate on attacks
All 30 attack queries (15 regex + 15 jailbreak) blocked correctly. Zero false negatives across the full eval suite.
🎭
Semantic attacks caught at L2
Rewording, euphemisms, and indirect escalation that bypass L1 regex are caught by the LLM classifier with >70% confidence.
🔒
RBAC via silence, not refusal
Public users querying admin-tier data receive no_docs — not a refusal message. The LLM cannot leak what it was never given.
⚡
L1 costs nothing
Regex validation runs in <1ms with zero LLM API calls. Obvious attacks are eliminated before any compute is spent.
🔗
Hybrid retrieval + reranking
Dense ChromaDB + sparse BM25 merged via RRF, then cross-encoder reranked. Precision-focused — only the most relevant role-filtered chunks reach the LLM.
📋
Full audit trail
Every query — attack or safe — logged with timestamp, layer reached, block reason, and response fingerprint. Layer 5 always runs.

Known Limitations
What this demo does not solve.
⚠️
No real authentication
The role selector is UI-only. Anyone can switch to admin. A production system would bind roles to verified session tokens.
🔄
Ephemeral ChromaDB
The vector store resets on HF Space restart. Click "Load Demo Data" again after each restart to rebuild the index.
🎭
Regex is bypassable
Layer 1 uses literal patterns. Unicode tricks, encoding obfuscation, or paraphrasing can slip through — that's why Layer 2 (LLM classifier) exists.
📦
9 demo chunks only
The deployed Space ingests 9 illustrative chunks. A production system would index thousands from full document PDFs with proper chunking strategy.
